Part 1: Setting up the Kubernetes Cluster

Cluster Setup

Launch a Kubernetes Cluster of Ubuntu 22.04 with two nodes (one master, one worker) using the Cloudformation Template to Create Kubernetes Cluster.

📝 Note

Once the master node is up and running, the worker node automatically joins the cluster.

💡 Alternative

If you have issues with the Kubernetes cluster, you can use the playground at: killercoda.com/playgrounds

Verify Installation

Check if Kubernetes is running and nodes are ready:

bash 
kubectl cluster-info
kubectl get no

Part 2: Kubernetes Secrets

Creating Secrets Using kubectl

Secrets contain sensitive user credentials required by Pods to access databases or services.

Create Credential Files

bash 
# Create files for the example
echo -n 'admin' > ./username.txt
echo -n '1f2d1e2e67df' > ./password.txt
📝 Note

The -n flag prevents outputting a trailing newline.

Create Secret from Files

bash 
kubectl create secret generic db-user-pass \
  --from-file=./username.txt \
  --from-file=./password.txt

Using Custom Key Names

bash 
kubectl create secret generic db-user-pass-key \
  --from-file=username=./username.txt \
  --from-file=password=./password.txt
⚠️ Special Characters

Special characters like $, \, *, =, and ! require escaping. Use single quotes:

kubectl create secret generic dev-db-secret \
  --from-literal=username=devuser \
  --from-literal=password='S!B\*d$zDsb='

Creating Secrets Manually

You can create Secrets using YAML manifests with base64-encoded values.

Encode Values to Base64

bash 
echo -n 'admin' | base64
# Output: YWRtaW4=

echo -n '1f2d1e2e67df' | base64
# Output: MWYyZDFlMmU2N2Rm

Create Secret YAML

yaml 
apiVersion: v1
kind: Secret
metadata:
  name: mysecret
type: Opaque
data:
  username: YWRtaW4=
  password: MWYyZDFlMmU2N2Rm

Using Secrets in Pods

Inject secrets into pods as environment variables:

yaml 
apiVersion: v1
kind: Pod
metadata:
  name: secret-env-pod
spec:
  containers:
  - name: mycontainer
    image: redis
    env:
      - name: SECRET_USERNAME
        valueFrom:
          secretKeyRef:
            name: mysecret
            key: username
      - name: SECRET_PASSWORD
        valueFrom:
          secretKeyRef:
            name: mysecret
            key: password
  restartPolicy: Never

Part 3: ConfigMaps in Kubernetes

What is a ConfigMap?

ConfigMaps allow you to decouple configuration from container images, making applications more portable.

Creating ConfigMaps

From Literal Values

bash 
kubectl create configmap demo-config --from-literal=greeting=Hola

From YAML File

yaml 
apiVersion: v1
kind: ConfigMap
metadata:
  name: demo-config
data:
  greeting: Hola

Using ConfigMaps in Applications

yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: demo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: demo
  template:
    metadata:
      labels:
        app: demo
    spec:
      containers:
        - name: demo
          image: ondiacademy/demo:hello-config-env
          ports:
            - containerPort: 8888
          env:
            - name: GREETING
              valueFrom:
                configMapKeyRef:
                  name: demo-config
                  key: greeting

Using ConfigMaps as Volumes

Mount ConfigMaps as files inside containers:

yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:latest
          ports:
            - containerPort: 80
          volumeMounts:
          - mountPath: /usr/share/nginx/html/
            name: nginx-config-volume
            readOnly: true
      volumes:
      - name: nginx-config-volume
        configMap:
          name: nginx-config
          items:
          - key: content
            path: index.html

Best Practices

  • Always use Secrets for sensitive data (passwords, tokens, keys)
  • Use ConfigMaps for non-sensitive configuration
  • Never commit secrets to version control
  • Consider using external secret management tools for production
  • Regularly rotate secrets
  • Use RBAC to control access to secrets

Bölüm 2: Kubernetes Secrets

kubectl Kullanarak Secret Oluşturma

Secret'lar, Pod'ların veritabanlarına veya servislere erişmek için ihtiyaç duyduğu hassas kullanıcı kimlik bilgilerini içerir.

Kimlik Bilgisi Dosyaları Oluşturma

bash 
# Örnek için dosyalar oluştur
echo -n 'admin' > ./username.txt
echo -n '1f2d1e2e67df' > ./password.txt
📝 Not

-n flag'i satır sonu karakterinin çıktıda yer almamasını sağlar.

Dosyalardan Secret Oluşturma

bash 
kubectl create secret generic db-user-pass \
  --from-file=./username.txt \
  --from-file=./password.txt

Özel Key İsimleri Kullanma

bash 
kubectl create secret generic db-user-pass-key \
  --from-file=username=./username.txt \
  --from-file=password=./password.txt
⚠️ Özel Karakterler

$, \, *, = ve ! gibi özel karakterler escape edilmelidir. Tek tırnak kullanın:

kubectl create secret generic dev-db-secret \
  --from-literal=username=devuser \
  --from-literal=password='S!B\*d$zDsb='

Manuel Secret Oluşturma

YAML manifest'ler kullanarak base64-encoded değerlerle Secret oluşturabilirsiniz.

Değerleri Base64'e Encode Etme

bash 
echo -n 'admin' | base64
# Çıktı: YWRtaW4=

echo -n '1f2d1e2e67df' | base64
# Çıktı: MWYyZDFlMmU2N2Zm

Secret YAML Oluşturma

yaml 
apiVersion: v1
kind: Secret
metadata:
  name: mysecret
type: Opaque
data:
  username: YWRtaW4=
  password: MWYyZDFlMmU2N2Zm

Pod'larda Secret Kullanımı

Secret'ları environment variable olarak pod'lara inject edin:

yaml 
apiVersion: v1
kind: Pod
metadata:
  name: secret-env-pod
spec:
  containers:
  - name: mycontainer
    image: redis
    env:
      - name: SECRET_USERNAME
        valueFrom:
          secretKeyRef:
            name: mysecret
            key: username
      - name: SECRET_PASSWORD
        valueFrom:
          secretKeyRef:
            name: mysecret
            key: password
  restartPolicy: Never

Bölüm 3: Kubernetes'te ConfigMaps

ConfigMap Nedir?

ConfigMap'ler, yapılandırmayı container image'larından ayırmanıza olanak tanır ve uygulamaları daha taşınabilir hale getirir.

ConfigMap Oluşturma

Literal Değerlerden

bash 
kubectl create configmap demo-config --from-literal=greeting=Hola

YAML Dosyasından

yaml 
apiVersion: v1
kind: ConfigMap
metadata:
  name: demo-config
data:
  greeting: Hola

Uygulamalarda ConfigMap Kullanımı

yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: demo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: demo
  template:
    metadata:
      labels:
        app: demo
    spec:
      containers:
        - name: demo
          image: ondiacademy/demo:hello-config-env
          ports:
            - containerPort: 8888
          env:
            - name: GREETING
              valueFrom:
                configMapKeyRef:
                  name: demo-config
                  key: greeting

ConfigMap'leri Volume Olarak Kullanma

ConfigMap'leri container içinde dosya olarak mount edin:

yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx

  replicas: 1

    matchLabels:
      app: nginx

    metadata:

        - name: nginx

          - mountPath: /usr/share/nginx/html/
            name: nginx-config-volume

      - name: nginx-config-volume

En İyi Pratikler

  • Hassas veriler için her zaman Secret kullanın (password'ler, token'lar, key'ler)
  • Hassas olmayan yapılandırma için ConfigMap kullanın
  • Secret'ları asla version control'e commit etmeyin
  • Production için external secret management tool'ları düşünün
  • Secret'ları düzenli olarak rotate edin
  • Secret'lara erişimi kontrol etmek için RBAC kullanın
